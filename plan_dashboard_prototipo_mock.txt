PLAN DE TRABAJO - CAPA DASHBOARD (PROTOTIPO CON CAPAS SIMULADAS)

Objetivo general
- Diseñar y validar un dashboard funcional de la app de pronostico RUL, simulando las capas previas (datos, modelo, agente) sin implementarlas todavia.

Alcance
- Construir un prototipo navegable del dashboard.
- Simular respuestas de inferencia, riesgo y recomendaciones.
- Validar experiencia de usuario, flujo operativo y criterios de aceptacion UI.

Supuestos
- No existe aun integracion real con modelo RUL.
- No existe aun API real de backend.
- Se usaran datos mock y servicios simulados.

Precondiciones
1) Activar entorno:
   conda activate cmapss
2) Instalar libreria de prototipado (si no esta):
   pip install streamlit plotly pandas
3) Crear carpetas:
   dashboard/
   dashboard/mock/
   dashboard/assets/
   out/dashboard/

FASE 1 - Definicion funcional del dashboard
Objetivo
- Precisar que pantallas, widgets y decisiones debe soportar el MVP.

Tareas
1) Definir historias de usuario principales:
   - operador carga datos de motor
   - sistema muestra RUL estimado
   - sistema clasifica nivel de riesgo
   - sistema sugiere accion de mantenimiento
2) Definir vistas:
   - Vista resumen (KPI principales)
   - Vista detalle de motor
   - Vista tendencias historicas
   - Vista recomendaciones
3) Definir estados visuales:
   - normal / warning / critico
   - cargando / error / sin datos

Entregables
- out/dashboard/01_user_flows.txt
- out/dashboard/01_screen_map.txt

Criterio de salida
- Mapa de navegacion y casos de uso aprobados.

FASE 2 - Contratos mock de entrada/salida
Objetivo
- Simular interfaces de las capas previas para desacoplar UI del backend real.

Tareas
1) Definir schema de input simulado:
   - unit_id, cycle, op_settings, sensores
2) Definir schema de output simulado:
   - rul_pred
   - risk_level
   - confidence_band
   - recommendation_text
3) Definir 3 perfiles mock:
   - motor saludable
   - motor en degradacion media
   - motor critico

Entregables
- dashboard/mock/input_schema.json
- dashboard/mock/output_schema.json
- dashboard/mock/mock_predictions.json

Criterio de salida
- Contratos mock estables para consumir desde UI.

FASE 3 - Implementacion del prototipo UI
Objetivo
- Construir un dashboard ejecutable con datos simulados.

Tareas
1) Crear app principal:
   - dashboard/app.py
2) Componentes minimos:
   - panel de ingreso (manual o archivo CSV)
   - card de RUL estimado
   - semaforo de riesgo
   - grafico de tendencia temporal
   - bloque de recomendaciones
3) Navegacion por pestañas:
   - Resumen
   - Detalle
   - Historico
   - Recomendaciones

Entregables
- dashboard/app.py
- dashboard/components.py
- dashboard/mock/service.py

Criterio de salida
- Prototipo navegable, sin errores y con flujo completo simulado.

FASE 4 - Simulacion de comportamiento dinamico
Objetivo
- Hacer que el dashboard responda como si el modelo real estuviera operativo.

Tareas
1) Simular latencia de inferencia (ej. 300-1200 ms).
2) Simular variacion de prediccion por ciclo.
3) Simular errores controlados:
   - datos incompletos
   - valores fuera de rango
4) Mostrar mensajes de estado claros al usuario.

Entregables
- dashboard/mock/scenarios.json
- out/dashboard/04_state_handling_notes.txt

Criterio de salida
- UI robusta frente a escenarios normales y de falla.

FASE 5 - Validacion UX y criterios de aceptacion
Objetivo
- Evaluar si el dashboard cumple necesidades de uso antes de integrar backend real.

Tareas
1) Checklist funcional:
   - carga de datos
   - prediccion visible
   - riesgo visible
   - recomendacion visible
2) Checklist de usabilidad:
   - legibilidad
   - tiempos percibidos
   - claridad de alertas
3) Ajustar textos, colores y jerarquia visual.

Entregables
- out/dashboard/05_qa_checklist.txt
- out/dashboard/05_feedback_log.txt

Criterio de salida
- MVP aprobado para conectar capas reales en siguiente iteracion.

FASE 6 - Plan de transicion a integracion real
Objetivo
- Dejar preparado el reemplazo de mocks por servicios reales.

Tareas
1) Identificar puntos de acoplamiento mock->real.
2) Definir interfaz unica de servicio de inferencia.
3) Plan de reemplazo incremental:
   - paso 1: conectar preprocesamiento real
   - paso 2: conectar modelo real
   - paso 3: conectar recomendaciones reales
4) Definir pruebas de regresion UI despues de cada conexion.

Entregables
- out/dashboard/06_transition_plan.txt
- out/dashboard/06_integration_risks.txt

Criterio de salida
- Ruta clara para pasar de prototipo mock a producto funcional.

Orden recomendado
1) Fase 1
2) Fase 2
3) Fase 3
4) Fase 4
5) Fase 5
6) Fase 6

Tiempo estimado (1 persona)
- Fase 1-2: 0.5 a 1 dia
- Fase 3-4: 1 a 2 dias
- Fase 5-6: 0.5 a 1 dia
- Total: 2 a 4 dias

Definicion de exito
- Existe un dashboard funcional con experiencia completa de uso.
- La UI esta desacoplada de backend real mediante contratos mock.
- El equipo puede validar flujo de negocio antes de implementar capas previas.
