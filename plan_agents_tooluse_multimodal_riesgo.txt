PLAN DE TRABAJO - CAPA "AGENTS, TOOL USE Y MULTIMODAL EXTRACTION" (V1 MEJORADA)
ENFOQUE: LOGICA DE RIESGO Y RECOMENDACIONES (PHM C-MAPSS)

Objetivo general
- Disenar e implementar una capa inteligente que convierta senales de degradacion en:
  1) nivel de riesgo interpretable
  2) recomendacion operativa accionable
  3) trazabilidad de por que se emitio esa recomendacion
- Mantener desacople de capas para que el dashboard (capa 3) siga operativo con payloads mock.

Principio de arquitectura (contract-first)
- Las capas se desarrollan de forma independiente.
- Lo critico son los contratos entre capas, no la implementacion interna.
- Capa 3 debe consumir contrato estable aunque capa 2 evolucione internamente (reglas, LLM o hibrido).

Contexto
- Capa 1: pronostico RUL (modelo predictivo)
- Capa 2: agents + tool use + multimodal (riesgo y recomendaciones)
- Capa 3: dashboard (front)

Contrato de entrada de capa 2
- rul_pred
- confidence_band
- dataset_id, unit_id, cycle
- op_settings (3), sensors (21), source
- history (opcional)

Contrato de salida de capa 2
- risk_level
- risk_score
- recommendation_text
- recommendation_priority
- recommendation_alternatives
- rationale
- audit_record_id
- evidence_summary
- service_status (ok/degraded/fallback)

Precondiciones
1) Entorno activo:
   conda activate cmapss
2) Estructuras base:
   src/agent_layer/
   out/agent_layer/
3) Contratos de dashboard disponibles y versionados.

FASE 1 - Contrato y gobernanza de decisiones
Objetivo
- Formalizar I/O y criterios de seguridad antes de automatizar recomendaciones.

Tareas
1) Definir schema de entrada y salida (v1).
2) Versionar contratos y ejemplos validos/invalidos.
3) Definir politicas de decision:
   - nunca ocultar incertidumbre
   - no emitir acciones de alto impacto sin evidencia minima
   - priorizar recomendaciones conservadoras ante ambiguedad
4) Definir catalogo inicial de recomendaciones y severidad.

Entregables
- out/agent_layer/01_input_contract_v1.json
- out/agent_layer/01_output_contract_v1.json
- out/agent_layer/01_contract_examples.json
- out/agent_layer/01_policy_rules.txt
- out/agent_layer/01_recommendation_catalog.csv

Criterio de salida
- Contrato y reglas aprobadas por el equipo.

FASE 2 - Motor de riesgo base (reglas + score)
Objetivo
- Implementar una version robusta y explicable de clasificacion de riesgo.

Tareas
1) Construir score de riesgo con factores:
   - RUL estimado
   - ancho de confidence band
   - tendencia reciente de degradacion
   - consistencia entre senales
2) Definir mapeo score -> risk_level.
3) Agregar reglas hard-stop:
   - critical inmediato si RUL debajo de umbral extremo
   - warning preventivo si incertidumbre alta con degradacion rapida
4) Generar rationale estructurado por regla activada.

Entregables
- src/agent_layer/risk_engine.py
- out/agent_layer/02_risk_scoring_design.txt
- out/agent_layer/02_thresholds_config.json

Criterio de salida
- Risk level reproducible y explicable para casos de prueba.

FASE 3 - Agent orchestration y Tool Use
Objetivo
- Coordinar herramientas para enriquecer decision y recomendacion.

Tareas
1) Definir agente orquestador con pipeline:
   ingest -> validate -> enrich -> score -> recommend -> audit
2) Integrar tool use minimo:
   - tool A: lector de predicciones/model output
   - tool B: consulta de reglas y catalogo de acciones
   - tool C: generador de resumen explicable para dashboard
3) Definir fallback si alguna herramienta falla:
   - continuar con reglas base
   - marcar salida con bandera de degradacion de servicio
4) Registrar auditoria de cada inferencia.

Entregables
- src/agent_layer/orchestrator.py
- src/agent_layer/tools.py
- out/agent_layer/03_toolchain_contracts.txt
- out/agent_layer/audit_log.jsonl

Criterio de salida
- Orquestacion estable con degradacion controlada.

FASE 4 - Multimodal extraction para evidencia contextual
Objetivo
- Incorporar texto/tablas/imagenes como evidencia adicional para riesgo.

Tareas
1) Definir tipos multimodales permitidos:
   - PDF tecnico
   - tabla CSV/XLSX
   - imagen de inspeccion (si aplica)
2) Implementar extractor de evidencia:
   - extraer entidades (componente, fallo, severidad, fecha)
   - normalizar a formato comun
3) Enlazar evidencia al rationale de riesgo.
4) Definir score de confianza de evidencia extraida.

Entregables
- src/agent_layer/multimodal_extractor.py
- out/agent_layer/04_evidence_schema.json
- out/agent_layer/04_extraction_rules.txt
- out/agent_layer/04_tool_interfaces.txt

Criterio de salida
- Evidencia multimodal integrada sin romper el contrato v1.

FASE 5 - Generacion de recomendaciones accionables
Objetivo
- Traducir riesgo a acciones concretas de mantenimiento.

Tareas
1) Definir plantillas por nivel de riesgo y contexto.
2) Agregar priorizacion por ventana operativa:
   - next window
   - immediate inspection
   - monitor only
3) Incluir recomendaciones alternativas cuando incertidumbre sea alta.
4) Adjuntar justificacion corta y larga para UI/reportes.

Entregables
- src/agent_layer/recommender.py
- out/agent_layer/05_recommendation_templates.txt
- out/agent_layer/05_priority_matrix.csv

Criterio de salida
- Recomendaciones consistentes, accionables y auditables.

FASE 6 - Integracion con dashboard y auditoria
Objetivo
- Mostrar decision completa en UI y dejar huella auditable.

Tareas
1) Extender payload consumido por dashboard:
   risk_score, priority, rationale, audit_record_id, service_status.
2) Actualizar pestana Recomendaciones con:
   - accion principal
   - alternativas
   - evidencias clave
3) Agregar vista de "por que esta decision".
4) Verificar compatibilidad con mocks de prueba.

Entregables
- out/agent_layer/06_dashboard_mapping.txt
- out/agent_layer/06_explainability_checklist.txt
- out/dashboard/contract_integration_checklist.txt

Criterio de salida
- Dashboard comunica riesgo + recomendacion + explicacion con contrato estable.

FASE 7 - Uso de LLM real (decision e incorporacion controlada)
Objetivo
- Definir donde conviene usar LLM real sin comprometer control, costo ni seguridad.

Tareas
1) Evaluar opciones de motor interno:
   - reglas + score (deterministico)
   - reglas + LLM (hibrido recomendado)
   - LLM dominante con guardrails
2) Definir tareas candidatas para LLM:
   - redaccion de recommendation_text
   - enriquecimiento de rationale
   - resumen ejecutivo de evidencia
3) Definir configuracion operacional:
   - proveedor/modelo
   - manejo de secretos (ejemplo: GEMINI_API_KEY en entorno local y Streamlit Cloud)
   - politicas de timeout/reintento/limite de costo
4) Definir fallback obligatorio:
   - si LLM falla, responder con reglas base y service_status=fallback.

Entregables
- out/agent_layer/07_engine_decision_record.txt
- out/agent_layer/07_llm_integration_policy.txt
- out/agent_layer/07_nonfunctional_requirements.txt

Criterio de salida
- LLM incorporado solo donde agrega valor, sin romper contrato ni operacion.

FASE 8 - Validacion y hardening
Objetivo
- Probar seguridad, consistencia y utilidad de la capa.

Tareas
1) Casos nominales:
   healthy, warning, critical.
2) Casos adversos:
   input incompleto, ruido extremo, conflicto entre fuentes.
3) Pruebas de contrato y regresion UI.
4) Definir criterios Go/No-Go para produccion interna.

Entregables
- out/agent_layer/08_test_matrix.txt
- out/agent_layer/08_failure_modes.txt
- out/agent_layer/08_acceptance_criteria.txt

Criterio de salida
- Capa validada para operar como componente de decision en la app.

Definicion de exito

- Capa 3 permanece estable aunque capa 2 evolucione internamente.
- LLM real se usa de forma selectiva, con fallback seguro y control operacional.
